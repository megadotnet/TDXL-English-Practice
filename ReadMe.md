# TDXL English Practice

An all-in-one English learning and practice platform combining simulated exam environments, interactive speech-synthesis reading aids, vocabulary flashcards, and an AI-generated essay repository for comparative study.

🌐 **Language Versions:**
* **English** (Current Document)
* [简体中文 (Simplified Chinese)](ReadMe-ZhCn.md)

## 1. Project Overview

**TDXL English Practice** is designed to provide users with a comprehensive environment to master English skills (speaking, reading, vocabulary, writing, and exam-taking). The project is architected with a hybrid model, balancing a lightweight static portal with a high-performance, interactive database-driven Next.js application:

1. **Static Portal Core (Root level)**: A series of highly optimized, offline-friendly, single-page web applications built on vanilla HTML5, CSS3, and JavaScript. They utilize advanced Web APIs (such as Web Speech Synthesis and Range APIs) to provide features like paragraph reading with real-time word highlighting, vocabulary flashcards, and a structured 5-day writing plan.
2. **Advanced Mock Exam System (`Mock-exam-timer/GLM-Agent-timer`)**: A premium, full-stack Next.js application incorporating custom settings, multiple exam templates, a highly interactive radial timer dashboard, background ambient white noise player, scratchpad notes, session recovery state, and persistent test run analytics backed by a SQLite database.

## 3. Environment Dependencies & Requirements

To avoid runtime collisions, satisfy these baseline system requirements before building:

### 3.1 Software Requirements
* **Node.js**: **Runtime Version >= `v18.18.0`** (Recommended: **`v20.11.0 LTS`** or higher).
* **npm**: **Package Manager Version >= `v9.0.0`** (Recommended: **`v10.2.0`** or higher).
* **SQLite3 Library**: Pre-bundled with standard Node packages; system-level installs are not strictly required as Prisma packages its own engine binary.

### 3.2 Client-side Browser Support
* **Standard Modern Browser**: Google Chrome (>= 110), Microsoft Edge (>= 110), Safari (>= 16), Firefox (>= 110).
* **Web Speech API**: Required for TTS text reading and speech output. *Native (Local) system voice packages must be installed on the host OS for optimal real-time word highlighting.*

## 4. Local Deployment & Startup Steps

You can set up and run both the static portal and the advanced application locally using the scripts below:

### 4.1 Running the Static Portal
Since the root modules consist of vanilla HTML, CSS, and JS, they can be opened directly or served via any lightweight HTTP server:
* **Option A (Double-Click)**: Open [index.html](file:///g:/Projects/TDXL-English-Practice/index.html) directly in any modern web browser.
* **Option B (Local HTTP Server)**:
  ```bash
  # Using Node.js npx serve (recommended for accurate Web Speech behavior)
  npx serve -p 8080 .
  ```
  Then, navigate your browser to `http://localhost:8080`.

### 4.2 Deploying the Next.js Mock Exam Application
Follow these commands in your shell to build and start the database-backed Next.js application:

#### Step 1: Navigate to the Application Subdirectory
```bash
cd Mock-exam-timer/GLM-Agent-timer
```

#### Step 2: Install Package Dependencies
```bash
npm install
```

#### Step 3: Configure Environment Variables
Create a `.env` file in the `Mock-exam-timer/GLM-Agent-timer` directory (a default configurations file exists as `.env` by default):
* **On Windows (PowerShell)**:
  ```powershell
  New-Item -Path .env -ItemType File -Value "DATABASE_URL=file:./dev.db" -Force
  ```
* **On macOS / Linux (Terminal)**:
  ```bash
  echo "DATABASE_URL=file:./dev.db" > .env
  ```

#### Step 4: Initialize the SQLite Database (Prisma Setup)
Push the relational schema onto your SQLite database and generate the Prisma Client:
```bash
# Push schema changes to SQLite and create DB tables
npm run db:push

# Generate the typesafe Prisma Client compiler
npm run db:generate
```

#### Step 5: Start the Development Server
```bash
npm run dev
```
Open your browser and navigate to `http://localhost:3000`.

#### Step 6: Build & Run in Production Mode (Optional but Recommended for Testing)
Ensure your build is error-free and compile assets into a standalone server:
```bash
# Compile TypeScript, React components, and static build outputs
npm run build

# Start the optimized Node.js standalone production server
npm run start
```
The server will boot up in production mode, listen on port `3000`, and route requests via the compiled `.next/standalone/server.js` wrapper.

## 6. Development Standards & Guidelines

Maintain codebase integrity and smooth deployments by adhering to these development guidelines:

### 6.1 Coding Style & Linting
* **TypeScript Strict Mode**: Always define explicit interfaces or types. Avoid raw `any` declarations.
* **Component Modularity**: Keep page-level components simple by extracting complex logic into smaller sub-components (e.g., place specific panels under `src/components/timer/`).
* **ESLint Verification**: Run `npm run lint` before committing any updates to prevent standard formatting compilation failures.

### 6.2 CSS & Layout Guidelines
* **Tailwind v4 Best Practices**: Standard styling should rely strictly on utility tokens. Do not write duplicate inline CSS.
* **Responsive Layouts**: Design for a "Mobile-First" flow. Every element must adapt gracefully from small screens up to massive desktop screens. Utilize responsive modifiers (`sm:`, `md:`, `lg:`) to control flexible padding and dimensions.

### 6.3 State & API Interaction Rules
* **UI Local State**: Use standard React `useState` / `useMemo` for local variables.
* **Timer Global State**: Use Zustand (`src/components/timer`) for states that govern global timers, active exam runs, and configuration states.
* **API Requests**: Run client requests through TanStack React Query (`useQuery`, `useMutation`) rather than ad-hoc `useEffect` calls, ensuring that database histories automatically update across tables and graphs.

## 7. Troubleshooting & FAQ

### Q1: The Next.js build fails during `shx` copy commands.
* **Cause**: `shx` commands might fail if folder paths are missing or locked by other developer utilities.
* **Fix**: Ensure standard write permissions are unlocked for the target folder. Running `npm install` again ensures `shx` binaries link correctly to your shell environment.

### Q2: Word highlighting is misaligned in `WritingReader.html`.
* **Cause**: Web Speech API's cloud voices (e.g., Google US English) sometimes stream data in chunks that prevent exact word boundary triggers (`onboundary` event is not fired properly or lacks accurate character index metrics).
* **Fix**: Select a **Local** voice (labeled as `(本地 - 支持高亮)` in the selector dropdown). Local voices rely on your local operating system engine (Windows Speech API or macOS CoreSpeech), which provides flawless character-level tracking.

### Q3: Prisma database errors: `Prisma Client has not yet been generated` or SQLite lock errors.
* **Cause**: The client bundle hasn't been compiled, or the local sqlite file is occupied by another process.
* **Fix**:
  1. Terminate any background development instances.
  2. Execute `npm run db:generate` to compile types.
  3. Run `npm run db:push` to ensure tables align with schemas.

### Q4: Port 3000 is already in use by another local development project.
* **Cause**: Standard Next.js instances run on port `3000` by default.
* **Fix**: Modify your launching script or start your server on an alternative port by running:
  ```bash
  npx next dev -p 3005
  ```
